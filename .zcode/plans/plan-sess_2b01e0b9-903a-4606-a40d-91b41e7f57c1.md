## Plan: Eliminate "Log in" flash for connected users

### Root cause (3 compounding issues)
1. `ConnectWallet.tsx` renders a disabled "Log in" button while loading — visually identical to the real button, so users perceive a flash.
2. `useWalletGate` can't distinguish "has localStorage session, wagmi not yet hydrated" from "no session at all" — both surface as `loading: true, isConnected: false`.
3. `#nav-login-slot` is empty in SSR HTML until React mounts via portal (after the ~1.6 MB Providers chunk loads). Empty → "Log in" → real wallet address is the visible sequence.

### Fix — 4 layers, smallest first

**Layer 1 — `src/lib/wagmi/useWalletGate.ts`**
Expose `hasSession: boolean` (true when `sessionAddress` is set from localStorage). Refine `loading` so it's only true when we genuinely don't know yet: no session AND wagmi not ready. When we have a session but wagmi hasn't hydrated, keep `isConnected: false` but expose `hasSession: true` so callers render a placeholder rather than the connect button.

**Layer 2 — `src/components/app/ConnectWallet.tsx`**
- Replace the disabled "Log in" button (loading state for `nav` variant) with a non-clickable muted placeholder pill shaped like the connected wallet button.
- Add a new branch: `hasSession && !isConnected` → render the same placeholder pill (we know they're connected, just waiting for wagmi to confirm). Never render the "Log in" button while `hasSession` is true.

**Layer 3 — `src/components/global/Navigation.astro` + `MobileNav.astro`**
Server-render both visual states directly into `#nav-login-slot` / `#mobile-login-slot`, swapped by a tiny inline script that reads `document.documentElement.dataset.walletSession` (already set by the BaseLayout inline script before any React loads):
- `[data-slot-state="connected"]`: muted pill shaped like the connected wallet button ("Loading…")
- `[data-slot-state="anonymous"]`: the actual "Log in" button — only visible when no session

When React mounts and portals `ConnectWallet` in, it replaces whichever placeholder is showing, with no flash either way.

**Layer 4 — `src/components/app/WalletPanelPlaceholder.tsx`**
Add a `variant="button"` mode that renders a pill matching the nav button dimensions. Used by ConnectWallet for consistency with the existing placeholder pattern.

### Files touched

**Edit (5):**
- `src/lib/wagmi/useWalletGate.ts` — expose `hasSession`, refine `loading`
- `src/components/app/ConnectWallet.tsx` — placeholder branch during loading + `hasSession`
- `src/components/app/WalletPanelPlaceholder.tsx` — add `variant="button"` pill
- `src/components/global/Navigation.astro` — SSR placeholder in `#nav-login-slot` + inline swap script
- `src/components/global/MobileNav.astro` — same for `#mobile-login-slot`

No new files.

### Performance angle
The flash is mostly a perceived-load issue — the heavy Providers bundle blocks React mount. The SSR placeholder (Layer 3) is the load-faster fix: by rendering the correct visual state in raw HTML before any JS runs, the page *looks* loaded immediately. The actual hydration time is unchanged, but the user no longer sees empty → wrong → right.

### Verification
- `npm run build` for TypeScript
- Manual with saved session: hard-reload — should see muted placeholder pill instantly, never "Log in"
- Manual without session: hard-reload — should see "Log in" button immediately (no empty slot)
- Confirm React still hydrates and replaces placeholder with real wallet menu

### Risks
- `data-wallet-session` is already set by the existing BaseLayout inline script — swap logic reuses existing infrastructure, no new localStorage parsing in nav.
- `createPortal` replaces slot children cleanly; no duplicate buttons.
- If `privyAppId` is unset, SSR markup falls back to existing "Login unavailable" span (unchanged branch).
